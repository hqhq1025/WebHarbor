Webhook: booking.created

Receive a signed callback whenever a new reservation is confirmed on the Booking platform.

Event type

The webhook fires with event_type = booking.created. A separate booking.cancelled event is documented elsewhere.

Sample payload

{
  "event_type": "booking.created",
  "event_id": "evt_8b1e7e9c4d3a4f0e",
  "occurred_at": "2026-05-12T12:00:00Z",
  "booking": {
    "booking_number": "BKN-DEMO01",
    "property_slug": "the-plaza-new-york-new-york",
    "city_slug": "new-york",
    "check_in": "2026-06-12",
    "check_out": "2026-06-15",
    "total": 1820.00,
    "currency": "USD",
    "guest": {
      "first_name": "Sophie",
      "last_name": "Martin",
      "email": "sophie.m@test.com"
    }
  }
}

The property is identified by the property_slug field; the city by city_slug. Use both to disambiguate listings sharing a name across cities.

Signature

Every POST carries the header X-Booking-Signature, computed as HMAC-SHA256(secret, raw_body_bytes) hex-lowercase. Partners must recompute and compare in constant time before trusting any payload.

The signature timestamp is provided in X-Booking-Signature-Timestamp. Reject requests older than 5 minutes.

Acknowledgement

Respond with HTTP status 200 OK within 5 seconds. Any other status (including 2xx variants other than 200) is treated as a delivery failure.

Retry policy

Failed deliveries are retried up to 5 times, with exponential backoff: 30s, 2m, 10m, 30m, 2h. After the final retry the event is moved to a dead-letter queue accessible via the partner dashboard.

Test the endpoint

POST a JSON body to this same URL and the server echoes back the payload with an ack_status field, so partners can dry-run their handler.
